# Stonly QA Automation Strategy

## Decomposition

I treated the assignment as a depth exercise rather than a breadth exercise. Instead of trying to automate every profile setting, I focused on one editable workflow: the Profile Headline. It has enough behavior to exercise meaningful QA concerns - read, write, clear, persistence, cleanup, selectors, and failure handling - without adding unrelated complexity such as uploads or onboarding.

The suite is split into three layers:

- Live UI behavior tests validate the Headline flow through the real Stonly UI and treat the backend as a black box.
- Hybrid mock contract tests use the live authenticated app shell, then mock Headline-related API responses with Playwright routes.
- Visual tests use the mocked Profile page state and Playwright snapshots to catch layout regressions without mutating the review account.

The UI model is fixture-first. Specs consume scenario fixtures such as `headlineProfile` or `mockedHeadlineProfile`; they do not use raw `{ page }` or a public `{ ui }` fixture. The private UI root owns the Playwright page and lazily composes pages/components. Pages own navigation, components own meaningful UI chunks, and assertions stay in specs or assertion helpers.

If this grew to hundreds of tests with several engineers and frequent releases, I would split ownership by product area, keep each area behind public scenario fixtures, and add conventions for tags, data setup, and cleanup. I would also move more setup/teardown to supported APIs once available, separate smoke/regression/visual/live suites more aggressively, and keep CI tiers small enough that PR feedback remains useful.

## Selector Strategy

Selector priority is:

1. Roles, labels, and accessible names where they are stable.
2. Stable test hooks such as `data-cy` when accessible selectors are missing or ambiguous.
3. CSS selectors only inside wrappers, and only when no better contract exists.

In this suite, the Profile heading is selected by role, the Headline editor input is selected by label, and the editor is scoped to the stable `[data-cy="headlineDialog"]` dialog. The closed Headline row uses `[data-cy="userHeadline"]`, and repeated edit controls are scoped through the Headline row so the test does not accidentally click the Name row edit action.

The main thing I would ask the development team for is a clearer testability contract:

- Stable test ids for setting rows and actions, ideally unique per row.
- Accessible labels for all editable fields.
- Named dialogs with stable roles and labels.
- Alert or toast semantics for failed saves, for example `role="alert"`.
- Less generic repeated hooks than `data-cy="editButton"` when several rows expose similar actions.

The goal is not to make tests depend on implementation details. The goal is to give the UI explicit, stable contracts for user-observable behavior.

## Reliability And Speed

Authentication is reused through Playwright storage state generated by a setup project. This keeps normal runs faster and avoids logging in before every test. Config is JSON-based and selected with `STONLY_CONFIG`, so future environments or datasets can be added without changing test code.

Live tests normalize the Headline to a known baseline before and after each test. The write test uses a unique timestamped value, and the clear test restores state through fixture teardown. Because live tests mutate one shared review account, the live spec runs serially. Mock and visual tests do not mutate the account and can run in parallel.

The mock and visual suites intentionally still use real Stonly auth and the live app shell for this take-home. Without owning the app shell or its complete API contract, a fully offline mock would be brittle and misleading. The mocked boundary is the Headline API behavior under test, which is enough to demonstrate contract-style testing and negative backend scenarios while keeping the app UI realistic.

The Playwright config is CI-aware:

- Local runs use Playwright's default worker count and no retries unless overridden.
- CI uses one worker and one retry by default.
- Traces are retained on failure locally and on first retry in CI.
- Screenshots are captured on failure.
- Videos are retained on failure in CI.

Retries should not be used to hide instability. A retry pass is a signal to investigate flakiness, not proof that the suite is healthy.

## Scope Judgment

I deliberately did not automate:

- Signup and email verification, because they introduce external inbox dependencies, anti-abuse checks, timing issues, and account sprawl.
- Exhaustive User Settings coverage, because one workflow covered well is more valuable for this assignment than many shallow checks.
- Profile picture upload, because file chooser behavior, image processing, and CDN timing would add noise before the core architecture was proven.
- Full cross-browser and device coverage, because Chromium is enough for this first high-signal workflow.
- A broad accessibility audit, because that deserves dedicated checks and manual review beyond basic selector signals.
- Deep backend validation in live tests, because live UI tests should validate user-observable behavior and keep backend assertions in contract or API layers.

Manual testing is still the right call for exploratory behavior around settings, visual polish, ambiguous product decisions, image upload UX, accessibility review, and browser/device combinations that are not justified in the small automated suite.

## Bugs And Rough Edges

These are the notes I would file or discuss with the team:

### Failed Headline Save Has No Visible Error

Area: User Settings / Profile Headline

Steps:

1. Open Profile settings.
2. Edit Headline.
3. Save while the `PUT /api/v1/user` request fails.

Expected: The user sees a clear error and can retry or cancel without losing context.

Actual: In the observed mocked failure path, the dialog closes and the previous Headline remains visible. No visible error was detected.

Impact: Users may believe the change was saved or may not understand why their update disappeared.

### Repeated Generic Edit Hooks Require Careful Scoping

Area: User Settings rows

Several rows expose the same `data-cy="editButton"` hook. Tests can handle this by scoping to a row, but unique row/action hooks would make the UI easier and safer to automate.

### Hybrid Mock Tests Still Depend On Real Auth

Area: Test environment

The mock and visual suites intentionally use real auth and the live app shell, then mock only Headline API behavior. This is acceptable for this take-home, but a production-grade suite would benefit from better app-shell contracts, API fixtures, or a lower-level way to bootstrap authenticated state.

### Review Credentials Are Obfuscated, Not Secret

Area: Test configuration

The review credentials are Base64-obfuscated in `config/review.json` for reviewer convenience. This is not secret management. In a real delivery setup, credentials should come from CI secrets or a managed test identity system.

## Fitting Into Delivery

For a team shipping many times a day, I would keep the default PR feedback loop small:

- Run typecheck and the mock contract suite on every pull request.
- Upload Playwright reports, screenshots, traces, and videos on failure.
- Use tags such as `@negative`, `@happypath`, and `@visual` for focused local and CI runs.
- Run visual snapshots after merges to `main` in the pinned Playwright Docker image.
- Run live mutation tests as opt-in, scheduled, or pre-release checks because they touch shared account state.

The most important delivery principle is fast, actionable feedback. A failing test should tell a developer which user workflow regressed, what evidence was captured, and whether the failure is a product issue, a contract mismatch, or test infrastructure instability.
